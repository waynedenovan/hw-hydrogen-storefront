## Step-by-step resolution (hw-storefront-ui-node-docker)

1. Prisma migration `20260710083134_add_import_run_entity_disable_product_sync`:
   new models `ImportRun` (job row: status/phase/offset/counters/heartbeat),
   `ImportBatchFailure` (a batch that failed twice), `EntityDisable` (granular
   disable switches, keyed supplierName+level+value like MarkupRule); `Product`
   gains `lastPushedHash/lastPushedAt/disabledReason/disabledAt`.
2. docker-compose.yml: added `./suppliers:/app/suppliers` bind mount.
3. NEW app/services/import-run.server.js — server-side background job runner.
   One short POST starts the run; UI polls every 3s; tunnel failures can no longer
   touch the run; closing the tab doesn't stop it. Fresh admin client per batch via
   `unauthenticated.admin(shop)` (offline token + refresh — survives multi-hour
   runs). Batch-failure policy exactly per spec: throw → wait 30s → retry; second
   throw → record ImportBatchFailure + skip to next batch; 5 consecutive failures →
   run ends `failed`; after import+disable phases, a `retrying` phase replays every
   unresolved failure once. Runs interrupted by a server restart are detected via
   stale heartbeat (>2min) and are resumable from the persisted offset (safe —
   create-vs-update is re-derived from Prisma per product).
4. Incremental imports (suppliers.server.js): `computeProductHash` = sha256 of raw
   product+pricing+stock entries + resolved markup (+ IMPORT_HASH_VERSION). Batch
   skips a product when hash matches, row is active, and a shopifyProductId exists —
   skipping all ~6 GraphQL round trips. Hash persisted only after a fully clean push
   (update/create + publish + collection-add), so partial failures re-push next run.
   MarkupRule changes change the hash → affected products re-push. "Force full
   import" checkbox bypasses. Repeat no-change runs now take minutes, not hours.
5. Disable-instead-of-delete: `runFinalizeBatch` → `runDisablePass`. Stale products
   get Shopify status DRAFT (`setProductStatus`, replaces `productDelete`) + local
   `active:false` with reason `manual` | `deselected` | `supplier_removed`; stale
   collections are UNPUBLISHED (`publishableUnpublish`, validated via
   shopify-dev-mcp; replaces `collectionDelete`) with an EntityDisable row as the
   persisted notice. Products/collections gone from the supplier file get the
   "deleted on the supplier's side" notice (UI banner + Disabled items page).
   Brands/sub-collections/sub-cat have no Shopify object (vendor/metafields) —
   disabling one drafts its products + records the EntityDisable row. Re-selecting
   an import-disabled collection republishes it and clears the row; `manual` rows
   are never auto-cleared. Nothing in the supplier flow deletes anymore
   (deleteProduct/deleteCollection remain only in the explicit Reset Store utility).
6. Granular disable UI: Disable/Enable toggle + badge on every Collection/Brand/
   Sub-collection/Sub-cat row on the Import page; NEW route /app/disabled
   ("Disabled items" nav entry) lists all disable switches per level with re-enable,
   all inactive products with reason badges (incl. supplier-removed notices), and a
   product search to disable individual products.
7. app.import.jsx rewired: old `apply-supplier-import-batch`/`finalize-batch`
   intents replaced by `start-supplier-import` / `import-run-status` (3s poll) /
   `cancel-supplier-import` / `resume-supplier-import`; progress shows phase,
   counts, failed batches; Resume button on interrupted/failed runs.
8. Tests: suppliers.server.test.js extended to 48 (hash-skip matrix, disable
   reasons, collection unpublish/republish, entity-disable gating); NEW
   import-run.server.test.js (12 — retry/skip/5-strikes/revisit/cancel/resume/
   stale-heartbeat, deterministic via injected sleep); product.server.test.js adds
   setProductStatus + publishableUnpublish mutation shapes.
