## Resolution (VERSION 2026.07.24)

**Ask**: Wayne asked whether the storefront's search already supported the
operand list above, or whether it needed to be added.

**Finding**: Checked `app/lib/searchIndex.server.ts` (the custom weighted
search engine built in session 2607191820 — Shopify's own `search()`/
`predictiveSearch()` can't rank against this catalog's custom metafields, so
it's never used here). None of the listed operands worked. `tokenize()`
split on every non-alphanumeric character, so `"`, `&`, `|`, `*`, `?` were
all silently discarded as separators. The existing "phrase" bonus never
fired for quoted input either — it kept the literal quote characters in the
comparison string, and no product title contains a literal `"`.

Cross-checked against Shopify's own documented search grammar
(shopify.dev/docs/api/usage/search-syntax, used across storefront/admin/
GraphQL) via shopify-dev-mcp. Wayne's list closely mirrors it, with two
divergences flagged and then explicitly excluded per his direction:
Shopify's NOT is `NOT`/leading `-`, not a bare `not` keyword; Shopify has no
`..` range syntax (it uses comparators like `price:>100`) and no `?`
single-char wildcard at all (a Lucene/Solr convention).

**Decision (confirmed with Wayne before implementing)**: build `""`, `AND`/
`&`, `OR`/`|`, and `*` following Shopify's own grammar; keep `?` as a
deliberate custom addition; exclude `NOT` and `..` range entirely — not
built. Also confirmed: plain queries with no operator syntax at all must
keep today's exact implicit-OR ranked behavior (Shopify's true default is
AND-between-bare-terms, which would have silently reduced results for
everyday multi-word searches against this catalog's inconsistent
metafields — a regression risk on a search feature only just fixed from
being completely dead).

**Fix**: added `parseQuery()`/`matchesQuery()` (exported) to
`app/lib/searchIndex.server.ts`. `parseQuery` extracts quoted phrases first
(via a NUL-byte sentinel placeholder so they survive whitespace splitting),
then builds AND-groups of OR-groups mirroring Shopify's documented
precedence (OR binds tighter than AND — their own example
`bob OR norman AND Shopify` == `(bob OR norman) AND Shopify`). Wildcards
(`*`/`?`) compile to an unanchored regex (matches anywhere in a field, not
prefix-only like Shopify's stricter semantics — deliberate, since this
catalog's product codes like `AO757/0813` benefit from inline wildcards,
e.g. `AO757*0813`). `searchCatalog()` now branches on
`parsed.hasOperators`: false → completely unchanged legacy path; true →
boolean-gate via `matchesQuery` first, then rank the matched set through the
existing field-weighted scorer (NOT gated by `score > 0` in this branch,
since a wildcard/phrase match can legitimately score 0 against the
plain-text tokens used only for ranking — gating would silently drop valid
matches).

**Verification**: 24 vitest cases added (`app/lib/__tests__/searchIndex.server.test.ts`)
covering `parseQuery` unit behavior (incl. the Shopify precedence example
and symbol/keyword equivalence), `searchCatalog` operator integration, and
an explicit regression test locking in "no operators = unchanged". All
green; `npm run typecheck` clean for the touched files (pre-existing
unrelated `storefrontapi.generated` errors elsewhere in the repo are not
caused by this change). Rebuilt + recreated the container (no source
bind-mount) and verified live via Playwright/curl: exact phrase requires
adjacency (reversed-word phrase correctly returns zero results), `AND`
narrows to the true intersection, `OR`/`AND` symbol and keyword forms
produce identical result lists, `*`/`?` wildcards match inline (not
prefix-only) including through non-title fields, and the header predictive
dropdown (not just the SSR `/search` page) correctly reflects an `AND`
query live.
