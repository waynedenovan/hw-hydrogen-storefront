### 1. Xero invoicing — payment reflects, no invoice created

Debug-mode investigation: `docker logs hw-storefront-ui-node-docker-storefront-ui-1 --since <today>` around order #1063's ITN showed the real chain of failure:

```
[xero] Could not auto-create item for product ID PROD-9627031306471 — Xero createItems failed: {"response":{"statusCode":429,...,"x-rate-limit-problem":"concurrent",...}}
[xero] Could not auto-create item for product ID PROD-9627023573223 — Xero createItems failed: {...429 concurrent...}
[api/payment/notify] Xero invoice error: Xero createInvoices failed: Item code 'PROD-9627031306471' is not valid
```

Two compounding bugs in `hw-storefront-ui-node-docker/app/services/xero.server.js` `createInvoiceFromOrder()`:
1. The per-order line-item loop built each line via `Promise.all(lineItems.map(async (li) => ...))`, i.e. every product needing a new Xero item was auto-created **concurrently**. The file's own top-of-file comment already documents Xero's real limit ("60 calls/minute and 5 concurrent per tenant... all batch loops in this file run sequentially") but this specific hot-path loop (unlike every bulk/background job in the same file) never followed that rule — so any order with 2+ new products triggered Xero's 429 concurrent-rate-limit error.
2. Worse: when the product-legacyResourceId fallback's `createXeroItem()` call threw (for *any* reason, not just rate-limiting — network blip, Xero downtime, etc.), the `catch` block still did `itemCode = code;`, referencing a Xero item code that was never actually created. That poisoned line then made the whole `createInvoices` call fail with `Item code '...' is not valid` — turning one missing item into a total invoice failure for the entire order. The sibling SKU-based fallback right above it did NOT have this bug (it correctly leaves `itemCode` unset on failure).

Fix (`xero.server.js`): converted the `Promise.all(...map(async...))` block to a sequential `for` loop (matching every other batch operation in the file), and removed the erroneous `itemCode = code` assignment in the failure branch so a failed auto-create just omits `itemCode` from that line (non-blocking) instead of poisoning the whole invoice.

Verification: code-reviewed against the exact log evidence above; the fix directly addresses both the concurrency trigger and the failure-mode that turned it into a total invoice loss. **Not yet re-verified with a fresh live order** — completing a real checkout + PayFast payment to confirm the invoice now appears in Xero requires an actual purchase/payment, which this session's guardrails correctly blocked from being auto-driven (filling real checkout details and submitting a live payment needs Wayne's own hands-on confirmation, not an agent-initiated transaction). Recommend Wayne place one more real test order with 2+ different new products in the cart and confirm the Xero invoice now appears.

### 2. Cart Qty "defaults to 50, jumps to 390 then back to 50" — NOT a code bug

Live Playwright test on hoseworld.store (product: Mamba Rope Ski Rope 10kg 12mm, msq=340) showed the product page's Quantity selector correctly starts at 340 and the Add to Cart POST body correctly sends `quantity: 340`. But Shopify's own Cart API response came back with:
```
"warnings":[{"code":"MERCHANDISE_NOT_ENOUGH_STOCK","message":"Only 50 items were added to your cart due to availability."}]
```
i.e. this specific heavily-tested product currently only has 50 units actually available in Shopify's live inventory (likely drawn down by repeated test-order sessions since the 3740-available stock screenshot was taken), and Shopify silently truncates the quantity to what's in stock. The app's `($locale).cart.tsx` action handler only ever checked/logged `userErrors`, never `warnings`, so this real stock constraint was completely invisible to the customer — looking exactly like a random "resets to 50" bug.

The msq stepping logic itself (`ProductCard.tsx`, `CartLineItem.tsx`, `($locale).products.$handle.tsx`, `QuantitySelector.tsx`) was already correct (fixed in the 2607141430 session) and needed no changes.

Fix: surfaced Shopify's `warnings` (not just `userErrors`) next to the Add to Cart button on both the Product Card (`ProductCard.tsx`) and Detailed Product Card (`($locale).products.$handle.tsx`), so a stock-capped quantity now shows "Only N items were added to your cart due to availability." instead of silently under-adding. New `.cart-stock-warning` style in `app.css`.

Separate (non-code) action recommended: if 3740 really should be available for this product, re-run the supplier stock import/sync so Shopify's live inventory matches — that's a data/ops step, not a code fix.

Verified live via Playwright: adding 340 to cart now shows "Only 50 items were added to your cart due to availability." directly under the button.

### 6. Detailed Product Card formatting — root cause found (recurring bug, finally fixed at the source)

This is why it "kept recurring despite being marked fixed" in prior sessions: every previous fix only ever touched `.product-main`/`.product-layout` sizing — never the actual cause, a **sitewide** rule in `app/styles/reset.css`:

```css
h1, h2, h3, h4, h5 {
  background: rgba(50, 50, 50, 0.85);
  padding: 0.5rem 0.75rem;
  border-radius: 6px;
  width: fit-content;
  ...
}
```

This blanket rule gives *every* heading on the site its own floating dark "pill" box — correct for headings that sit directly on the raw sitewide hero background (`.page-bg`, e.g. the homepage's "Online" heading, `CollectionCard`'s own title), but **wrong** wherever a heading already sits inside another element that provides its own dark background: `.page-card` (product detail page's `<h1>` title, "Product Details"/"Specifications" `<h3>`s, the collection page's `<h1>` title) and `ProductCard.tsx`'s own explicit `rgba(50,50,50,0.85)` wrapper div around its `<h3>` title. The result was a redundant nested box-within-a-box — most visible on long/wrapping product titles, which is exactly why it looked worse "when a new product is added" (a longer title wraps differently and makes the mismatched double box more obvious).

Fix: added a scoped override in `app.css` right after `.page-card--wide` — `.page-card h1..h5` and `.product-card h1..h5` reset `background/padding/border-radius/width` back to plain, matching the `collection-page-style` skill's actual rule ("headings inside a dark card are just white text, no extra box"). Left `reset.css`'s blanket rule untouched for every other bare heading (home page, `CollectionCard`, etc.) since those still need it for legibility against the raw background image — confirmed via Playwright that the homepage hero heading and collection-card titles still show their box correctly after this change.

Verified live via Playwright on both the product detail page (mobile 390px and desktop 1600px) and the collection page — the double-box is gone, heading renders as plain white text on the page-card, no regression on the homepage/collection-card headings that still legitimately need the box.
