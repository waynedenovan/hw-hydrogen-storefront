## Resolution — hw-hydrogen-storefront-node-docker Collection/Sub Collection footer (VERSION 2026.07.25)

**Scope note**: per "INCLUDE ONLY: GIT branch folders", this session only worked
inside this repo (hw-hydrogen-storefront-node-docker, branch
node-docker-migration). The hw-storefront-ui-node-docker items (import
Collection/Sub Collection field mapping, PayFast refund + Retry error +
Skill update) are a separate git repo/branch folder and were **not**
started — still open for a session run from that repo.

**MCP-Bridge check first (as required)**: `qdrant-find` on
hoseworld-dev-knowledge surfaced the full history of this exact symptom
class before any code was touched — session 2607081600 (dvh footer-gap
fix), and 2607081815, which explicitly noted at the time: "classless pages
(collections, policies…) keep their default white canvas" — i.e. Collection
pages were *deliberately* left out of the earlier footer fix, not
overlooked by accident, but the deliberate choice is exactly what produced
today's visible bug. shopify-dev-mcp was not consulted for the fix itself —
this is a React/CSS layout bug in this app's own PageLayout.tsx/reset.css,
not a Shopify Storefront API/GraphQL/theme pattern, so it's outside that
server's scope; the `collection-page-style` Skill *was* checked (see below)
since it's the documented styling reference for these routes.

**Root cause**: `app/components/PageLayout.tsx` computes a page-type flag
(`isHomePage`/`isProductPage`/`isAccountPage`/`isContactPage`/`isCartPage`)
for every other route and feeds it into the `.page-bg` wrapper's className
so `reset.css` can paint that page's own background — but there was no
`isCollectionPage` case. `app/routes/($locale).collections.$handle.tsx` and
`($locale).collections._index.tsx` therefore always rendered `.page-bg`
completely classless. `.page-bg` itself has no default background (only
`display:flex; flex-direction:column; flex:1 0 auto`), so it stayed fully
transparent, showing the browser's default white body canvas through any
leftover flex space below the (content-sized, non-stretching by design —
confirmed in session 2607072026) `.page-card`. The sticky-footer mechanism
itself (`body{display:flex;flex-direction:column;min-height:100dvh}` +
`.footer{margin-top:auto}`) was already correct and unaffected — the footer
was always at the true bottom of the document; the bug was purely visual:
a stark white gap between the dark `.page-card` and the black `.footer` on
any Collection/Sub Collection page short enough to leave flex space (e.g.
`/collections/ppe`, one sub-collection tile). Pages with enough products to
fill the viewport (e.g. `/collections/ra-safety`) never showed the gap,
which is why it read as inconsistent/intermittent rather than a flat CSS
mistake.

**Fix** (2 files):
- `app/components/PageLayout.tsx` — added `isCollectionPage` regex
  (`/^(\/(en-nz|en-au|en-us|en-za))?\/collections/`, matches both the
  collection index and `$handle` routes/sub-collections), wired into the
  SSR-rendered `.page-bg` className array (so it's correct on first paint,
  matching the existing pattern's own documented reasoning — no
  hydration-only flash of white), the `useEffect` body-class toggle/cleanup,
  and its dependency array.
- `app/styles/reset.css` — added `.page-bg.collection-page { background:
  var(--color-dark); }` (solid black, no image — Collection pages have no
  dedicated hero photo the way home/product/account/contact/cart do, and
  `--color-dark` is the exact same token `.footer` already uses, so the
  card now flows into the footer with zero visible seam regardless of
  content length) and added `body.collection-page` to the existing
  overscroll "black canvas" selector list, updating its comment (collection
  pages are no longer "classless").

**Why not a new background image**: considered giving Collection pages
their own hero photo like every other page type, but there is no
unused image asset in `public/images/` for it (`pumps-banner-1.jpg` is
already the sitewide `.header-hero` banner on every page, not free for
reuse here) and the `collection-page-style` Skill — the documented styling
reference for these exact routes — only specifies the `.page-card` card
tokens (rgba(50,50,50,0.65) wrapper, 80/90% max-width breakpoints, inner
rgba(50,50,50,0.85) overlays) and says nothing about a `.page-bg` hero
image. Picking new art would have been exactly the kind of undocumented
guess the task instructions say not to make; the solid-color fix is fully
justified by, and consistent with, what's already documented and already
live elsewhere in the codebase.
