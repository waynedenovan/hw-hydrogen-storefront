## RESOLUTION (2026-07-08, VERSION 2026.07.4)

### Debug-mode investigation (query-first per session instructions)
- `search_errors` (project_errors.db) and `qdrant-find` (MCP-Bridge) were checked first: no prior entries existed for a footer bottom-gap bug.
- Decoded `app/errors/260708153051-errors.png` (1918×1006) pixel-by-pixel (no PIL/imagemagick available in this environment, so a small zlib-based PNG decoder script was written ad hoc) to confirm the reported symptom was real, not a screenshot artifact: the solid-black footer band ends at y≈981, and a ~24px strip of the page's textured background image is visible below it down to y≈1005 (the true bottom of the screenshot).
- Live-tested the running container (`http://172.17.0.1:5130`, which is what `hoseworld.store` is tunneled to per the 2607080825 session) via Playwright MCP `browser_evaluate`/`getBoundingClientRect` across 1280×720, 1920×1000, 1918×876, and 390×700 viewports. In every case `footer.getBoundingClientRect().bottom` matched `document.documentElement.scrollHeight`/`window.innerHeight` exactly — no gap reproduced. The existing sticky-footer CSS (`body{display:flex;flex-direction:column;min-height:100vh}` in `app/styles/reset.css`, `.footer{margin-top:auto}` in `app/styles/app.css`) is structurally sound and was already working correctly for any fully-loaded, non-mobile-toolbar scenario.
- Conclusion: the gap is best explained by the well-known `100vh`-vs-actual-visible-viewport mismatch on mobile browsers with a collapsible address bar/toolbar (Playwright's viewport resize can't simulate real toolbar-collapse behavior, so it can't reproduce this class of bug directly). `app/styles/app.css`'s own SITE RULES require mobile support down to 360px, making this the relevant real-world path.

### Fix
1. `app/styles/reset.css` — added `min-height: 100dvh;` immediately after the existing `min-height: 100vh;` in the `body` rule, so browsers supporting dynamic viewport height use it (fixing the toolbar-collapse gap) while `100vh` remains as a fallback for browsers that don't support `dvh`.
2. Copied `media/images/rn-logo_50-150dpi.png` → `public/images/rn-logo_50-150dpi.png` (existing convention: curated site assets live in `public/images/`, source library in `media/images/`).
3. `app/components/Footer.tsx` — added a new `FooterBrandBar` component, rendered as the last child of `<footer>` after the existing `<FooterUtilities />`, so the pre-existing footer menu and policy-links row are unchanged. Left: `© {year} Hose World. All rights reserved.` Right: the rn-logo image followed by "Powered by: Rogue Nation" text (small, per task's "quite small" request). Uses a plain `<img>` tag matching `Header.tsx`'s own logo pattern (static local asset, not a Shopify CDN image, so the Hydrogen `<Image>` component doesn't apply).
4. `app/styles/app.css` — added `.footer-brand` (flex row, `justify-content:space-between`, `flex-wrap:wrap`, small font-size/opacity), `.footer-powered-by`, and `.footer-powered-logo` (18px tall) rules in the existing `components/Footer` section.
