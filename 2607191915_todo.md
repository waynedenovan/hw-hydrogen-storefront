## Question: implications of Shopify / Xero / PayFast being unavailable, and what redundancies exist

### 1. SHOPIFY unavailable — SEVERITY: CRITICAL (single point of failure for the whole storefront)

**Implications:**
- The storefront has no life without Shopify. The root loader awaits `HEADER_QUERY` with no guard (`app/root.tsx:104-118` — docstring says "the whole page should 400 or 500 error"), and `server.ts:145-149` turns any loader throw into a plain-text 500. Every page hard-fails.
- The cart lives IN Shopify — the session cookie holds only the cart id. During an outage the cart is unreadable (appears empty) but is NOT lost; it comes back when Shopify does.
- Account pages (orders, wishlist loader) call the Customer Account API unguarded → hard error.
- Admin app (hw-storefront-ui): OAuth sessions are stored locally (PrismaSessionStorage), so the app itself loads, but every Admin API operation fails. Import paths have retry helpers (`withNetworkRetry` 2 retries in product.server.js:873; `getPublicationIdsWithRetry` 3 attempts + the hard-abort guard from 2607191357) — payment-path Admin calls have NO retry.
- **WORST CASE — paid but no order**: if the Shopify Admin API is down at the moment PayFast's ITN arrives, `api.payment.notify.jsx` catches the failure, logs it, and still returns 200 to PayFast (`:180-182`, `:310-315` — deliberate, to stop ITN retries). The customer has paid; no Shopify order exists; `PayFastTransaction` is only written on success — so there is NO database record of the failure. Recovery = container logs + PayFast's own transaction history. This is the single biggest gap found.

**Redundancies in place:**
- Happy-eyeballs net-tuning (`server.ts:21-34`, and NODE_OPTIONS net-tuning in the ui compose) — protects against connect-level flakiness, not real outages.
- Deferred root data degrades gracefully: cart / login-state / footer queries all `.catch(() => null)`.
- Hydrogen's SWR sub-request cache + the new 5-minute search index cache can serve some already-cached content through brief blips.
- Local session storage keeps the admin app logged in through an outage.
- `restart: unless-stopped` on all containers.

**No redundancy for:** the storefront's critical render path (no cached-header fallback, no static maintenance page), and the ITN-during-outage scenario above.

### 2. XERO unavailable — SEVERITY: LOW customer-facing, MEDIUM back-office

**Implications:**
- Customers are unaffected — by design. Every Xero call in the payment path is explicitly non-blocking ("Xero failure must never block PayFast", notify.jsx:295-296): the Shopify order still completes and `PayFastTransaction` is persisted. What's lost is the invoice: `XeroInvoice` rows are only written on success, there is no retry queue, and `app.xero.invoices.jsx` has NO "create invoice for an existing order" action — an outage-orphaned order needs manual invoicing in Xero.
- Interactive admin Xero pages error (token refresh in `getAuthenticatedClient` xero.server.js:54-88 has no try/catch — a down token endpoint throws to the route).
- Returns are the model flow: `api.returns.request.jsx` persists the ReturnRequest row (PENDING) BEFORE any Xero work, so credit notes can be created after Xero recovers.

**Redundancies in place:** non-blocking design throughout the ITN; sequential rate-limit pacing (2200ms/1100ms delays) with one 60s-backoff retry in the bulk item job; `syncAllInvoiceStatuses` recovers stale invoice *statuses* (not missing invoices); manual per-invoice email button; XeroPendingChange approval queue; ReturnRequest persist-then-process.

### 3. PAYFAST unavailable — SEVERITY: HIGH during checkout, with a manual reconciliation fallback

**Implications:**
- Browsing/cart/checkout steps 1–3 unaffected. At "Proceed to Payment", `api.payment.initiate.jsx` creates the Shopify draft order and builds a *locally signed* redirect URL (no PayFast API call yet) — so with PayFast down the customer is redirected to an unreachable payment page. The draft order is orphaned (accumulates; cosmetic). The cart is preserved — `checkout.payment.tsx` returns 502/500 with an inline error and never touches the cart, and checkout/cancel correctly says "Your cart is still intact".
- Inbound direction: if PayFast (or our single cloudflared tunnel) can't deliver the ITN, payment completes but no order is created until an ITN arrives. Note the tunnel serves BOTH domains — one process is the delivery path for all ITNs (session 2607171535).
- No automatic alternative payment: `PUBLIC_PAYMENT_GATEWAY=shopify|payfast` exists as an env switch (Shopify hosted checkout), but it needs a config change + container restart, and it bypasses the VAT/draft-order/Xero flow — a manual contingency, not a failover.

**Redundancies in place:**
- `payfast.server.js` returns clean `{ok, error}` results and, crucially, the Transaction History API integration + `app.payfast.history.jsx` + `findTransactionByPfPaymentId()` constitute a working manual reconciliation tool for exactly the paid-but-no-order scenario.
- The error-surface on checkout is correct and non-destructive (customer can simply retry).

**No redundancy for:** proactive awareness — status.payfast.io's status API (documented in the payfast-status-api skill: `GET https://status.payfast.io/api/v2/status.json`, indicator none/minor/major/critical) is NOT integrated anywhere; no pre-payment availability check, no checkout banner during PayFast incidents.

### 4. Cross-cutting infrastructure findings
- No health checks, uptime monitoring, alerting, circuit breakers, or scheduled recovery jobs exist in either repo. The background job runner (jobs.server.js) is deliberately in-memory (re-runnable by design) — not a durable queue.
- SQLite DBs are bind-mounted to the host — all persisted state survives container redeploys.
- Single VPS + single cloudflared tunnel = shared infrastructure SPOF for storefront, admin app, and ITN delivery.

## Loss matrix (summary)

| Scenario | Outcome | Recoverable? |
|---|---|---|
| PayFast paid, Shopify Admin down at ITN | 200 to PayFast, no order, NO db record | Manual only (logs + PayFast history) — **worst gap** |
| PayFast paid, order created, Xero down | Order + transaction saved, no invoice, no queue | Manual invoice in Xero (no in-app re-create) |
| Xero down during return request | ReturnRequest persisted PENDING first | Yes — credit note later (model pattern) |
| Storefront API down | Site hard-500s; cart appears empty | Yes — transient; cart lives in Shopify |
| UI app down during payment initiate | Inline error, cart preserved | Yes — customer retries |
| PayFast web down at redirect | Orphaned draft order, cart preserved | Yes — retry when back |

## Recommendations (ranked, NOT implemented — for a future task)
1. **Persist-then-process for the ITN** (mirrors the ReturnRequest pattern): write a PayFastTransaction/failed-payment row with the raw ITN payload BEFORE calling `draftOrderComplete`, plus an admin "retry order completion" action. Eliminates the only permanent-loss path.
2. In-app **"create Xero invoice for order"** action (and/or an invoice retry pass over orders lacking an XeroInvoice row).
3. Guard the root `HEADER_QUERY` with a cached/static header fallback so a Shopify blip degrades the storefront instead of hard-500ing every page.
4. Pre-payment **PayFast status check** via status.payfast.io `/api/v2/status.json` (+ optional checkout banner during incidents).
5. Scheduled **reconciliation** comparing PayFast transaction history (service already exists) against local PayFastTransaction rows, flagging paid-but-orderless payments.
6. Basic alerting on ITN/invoice failures (currently console-log only).

## (1) ITN persist-then-process + admin retry — the permanent-loss path is closed
- New Prisma model `PayFastItn` (+ migration `20260719200500_add_payfast_itn_and_alert_event`, applied to the live bind-mounted DB): every COMPLETE ITN persisted with its raw payload BEFORE `draftOrderComplete` runs. Status lifecycle RECEIVED → COMPLETED | ORDER_FAILED | INVOICE_FAILED (+ MISSING_ORDER from reconciliation).
- Processing extracted from `api.payment.notify.jsx` into `app/services/payfastItn.server.js` (`recordItnReceived`, `processItn`, `retryItn`, shared `ORDER_DETAILS_QUERY`/`fetchOrderDetails`/`getAdmin`). notify.jsx is now a thin route: validate → persist → process → always 200 to PayFast (unchanged anti-retry semantics), duplicate ITNs for COMPLETED payments short-circuit.
- Idempotent retry: a row with an order id skips draft completion; an order with an existing XeroInvoice row skips invoicing; an "already completed" draft is recovered via a DraftOrderOrder probe query (covers the crash window between mutation and row update).
- Admin: new "Payment Log (ITN)" section at the top of `/app/payfast` with status badges, error text, and a Retry button for non-COMPLETED rows.
- **Verified live**: POSTed a fake sandbox ITN with an invalid draft id to the running container → endpoint returned 200 to "PayFast", and the DB now holds `TEST-ITN-2607191915 | ORDER_FAILED | draftOrderComplete userErrors: Draft order not found` plus a `payment-order-failed` AlertEvent. That exact scenario was previously unrecoverable (console log only). NOTE: this test row + its alert were left in the DB deliberately as a working demonstration — acknowledge the alert on the dashboard and use Retry/inspect as you like.

## (2) In-app "create Xero invoice for order"

## (3) Cached header fallback (hydrogen)
- `root.tsx` `loadCriticalData` now try/catches `HEADER_QUERY`: serves the module-level last-good header, or a static Hose World fallback (Header.tsx already falls back to `FALLBACK_HEADER_MENU` when menu is null). A Shopify blip now degrades the site (stale/minimal header, pages render) instead of hard-500ing every request. Verified the normal path live (menu renders, footer v2026.07.23).

## (4) Pre-payment PayFast status check (hydrogen)
- New `app/lib/payfastStatus.server.ts`: status.payfast.io `/api/v2/status.json` (per the payfast-status-api skill), 3s timeout, 60s TTL cache, **fails open** to 'unknown' when the status page itself is unreachable.
- `($locale).checkout.payment.tsx` blocks initiation with a friendly 503 only on indicator major/critical (before any draft order is created — no more orphaned drafts during a known outage). `($locale).checkout.tsx` Review step shows a warning banner from minor upward (`.checkout-payfast-status-banner` CSS, amber for minor / red for outage).

## (5) Scheduled PayFast-history reconciliation
- `app/services/payfastReconcile.server.js`: pulls PayFast weekly history, filters FUNDS_RECEIVED credits, and flags any payment with neither a PayFastTransaction row nor any PayFastItn row as `MISSING_ORDER` (+ alert) with an ITN-shaped replayable rawPayload — catching payments whose ITN never arrived at all (tunnel/app down).
- Triggers: "Reconcile vs PayFast history" button on `/app/payfast` (inline — completes in seconds), and hourly via `scripts/reconcile-cron.mjs` preloaded through compose NODE_OPTIONS (armed only when `ENABLE_PAYFAST_RECONCILE_CRON=true`, 5-min initial delay, unref'd timers) calling the new secret-authed `POST /api/payfast/reconcile`.
- **Verified live**: endpoint 401s without the X-Internal-Secret; with it, runs and returns a structured result. Current sandbox response is `ok:false "Merchant authorization failed"` — the PayFast **sandbox** merchant credentials don't have transaction-history API access; expected to work against live credentials (PAYFAST_SANDBOX=false). The scheduler logs `[reconcile-cron] armed — every 60min` on container start (multiple lines at boot are transient npm wrapper processes; only the long-lived server keeps a timer, and the run is idempotent anyway).

## (6) Basic failure alerting
- New Prisma model `AlertEvent` + `app/services/alerts.server.js` (`raiseAlert` — also emits a greppable `[ALERT]` docker-log line — `getUnacknowledgedAlerts`, `acknowledgeAlert`).
- Raised on: ORDER_FAILED, INVOICE_FAILED, MISSING_ORDER. Surfaced as a red banner at the top of the admin dashboard (`/app`) with per-alert Acknowledge buttons and deep links to the PayFast/Xero pages.
