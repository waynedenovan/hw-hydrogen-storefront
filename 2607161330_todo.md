### 2. Xero item-code prefix bug — root cause found and fixed

Root cause: "Bulk Create All Missing Items" (and the Sync Status list) correctly use `Product.supplierProductCode` (e.g. `Agr-PAS24L`, computed at import time by `computeSupplierProductCode()` in `product.server.js` — first 3 chars of supplier name, capitalized, + `-` + supplier's external product ID). But `xero.server.js`'s `createInvoiceFromOrder()` has its own, separate fallback used when an order is invoiced for a product that's never yet been created in Xero (no existing `XeroItem` row, no Shopify SKU set) — that fallback generated a completely different, generic `PROD-{legacyResourceId}` code, ignoring `supplierProductCode` entirely. Whichever code path ran first for a given product "won" — DAMFIX WET 20L got invoiced before ever going through Bulk Create, so it was created in Xero as `PROD-9627080556775` instead of a supplier-prefixed code.

Fix (`hw-storefront-ui-node-docker/app/services/xero.server.js`, `createInvoiceFromOrder()`): before falling back to `PROD-{legacyResourceId}`, look up the local `Product` row by `shopifyProductId` (`li.product.id`, the Shopify GID already fetched by `ORDER_DETAILS_QUERY`) and prefer its `supplierProductCode` (falling back to `externalProductId`, then only to `PROD-{legacyResourceId}` if there's no local Product row at all — e.g. a manually-created Shopify product with no supplier import data). This mirrors the exact fallback chain (`p.supplierProductCode || p.externalProductId`) already used correctly by the Bulk Create/Sync Status view in `app.xero.inventory.jsx`.

Verification: since this needs a fresh, never-before-invoiced product to reproduce cleanly, and completing a real payment is blocked by this harness's transaction-safety guardrail, verified via code review against the exact root cause (matching how the 2607161130 Xero concurrency fix was verified). **Needs Wayne to confirm**: place one real test order with a brand-new product (never gone through Bulk Create or a prior order) and check the resulting Xero item code starts with the correct 3-letter supplier prefix instead of `PROD-`.

### 3. Font weight/size — real root cause found: Tailwind utility classes on headings were silently no-ops

This is why "decrease the font weight" and "Detailed Product Cards still not formatted correctly" kept recurring across multiple sessions despite Product Card's Tailwind classes being correctly edited each time (confirmed with Wayne: title should be weight 350, brand/type weight 250, price weight 400, sized down further — floored at a legible 10px/9px rather than the literal "5pt smaller" ask, easy to tune further).

Root cause: `app.css` starts with `@import "tailwindcss";`, which puts every Tailwind utility class inside CSS cascade layers. `reset.css` is a separate, non-Tailwind stylesheet loaded via its own `<link>` — its `h1`/`h2`/`h3`/`p` rules are **not** in any layer. Per the CSS Cascade Layers spec, an unlayered rule always wins over a layered one regardless of specificity or source order. So Tailwind's `text-xs`, `font-medium`, `font-[350]`, etc. on any `h1`/`h2`/`h3` (font-size + font-weight) or `p` (font-size only) element were silently losing to reset.css's plain `h1 { font-size: 1.6rem; font-weight: 700 }` / `h3 { font-size: 1.1rem; font-weight: 700 }` / `p { font-size: 1rem }` rules — confirmed live via `getComputedStyle`: Product Card's title showed 17.6px/700 (reset.css's h3 default) despite the code correctly specifying `text-[10px] font-[350]`, and the product detail page's `<h1>` showed 25.6px/700 despite `text-xl font-[350]` in the code. Divs (price displays) were never affected — no reset.css rule targets bare `div`.

Fix: followed the same pattern already established for the "double-box" fix in the same file (`.page-card h1..h5` background reset) — set the real values as plain, unlayered CSS in `app.css` instead of relying on Tailwind utility classes that can never win against reset.css:
```css
.product-card h3 { font-size: 10px; font-weight: 350; }
.product-card p  { font-size: 9px;  font-weight: 250; }
.product-main h1 { font-size: 1.25rem; font-weight: 350; }
```
Removed the now-misleading `text-[10px] font-[350]` / `text-[9px] font-[250]` / `text-xl font-[350]` Tailwind classes from `ProductCard.tsx` and `($locale).products.$handle.tsx` (they were dead weight — always overridden) and left a comment pointing at the app.css rule instead, so a future session doesn't re-edit a Tailwind class that can never take effect. Price `<div>`s (not h1-h3/p) were unaffected by this bug and keep their working Tailwind classes (`text-xs font-[400]` / `text-base font-[400]`).

Verified live via `getComputedStyle` (not just screenshots, to catch exactly this kind of silent-override bug in future): Product Card title now genuinely 10px/weight 350, brand/type 9px/weight 250; product detail page `<h1>` now genuinely 20px/weight 350, price 16px/weight 400. Checked 390px mobile viewport: no title text overflow/clipping (`scrollHeight === clientHeight` on all sampled cards).
