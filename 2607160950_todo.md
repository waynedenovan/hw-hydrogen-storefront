### Issue B — Xero invoices silently stopped being created for every order since #1058 — FIXED

**Root cause (two compounding bugs, both in hw-storefront-ui-node-docker, both pre-dating this task but only now diagnosed):**

1. `app/services/xero.server.js` commit `9e29e65` (2026-07-05) added `cOGSAccountCode: COGS_ACCOUNT_CODE` to `createXeroItem()`'s `purchaseDetails` payload (create path, existing-item backfill path, and `batchFixSalesDetails()`) without ever setting the paired `inventoryAssetAccountCode`. Per `xero-node`'s own type docs, Xero derives "tracked inventory" status purely from whether `InventoryAssetAccountCode` + `COGSAccountCode` are *both* present — sending one without the other is invalid and Xero rejects every single item creation with `"Inventory Asset Account must be used with a Cost of Goods Sold Account."` (confirmed live in container logs for every product). This is caught non-blockingly per line-item inside `createInvoiceFromOrder()`, so it degrades invoices (no `itemCode`) but wasn't, by itself, the reason invoices stopped entirely.
2. `app/routes/api.payment.notify.jsx` (bug present since 2026-06-29, dormant until now): the `catch (xeroErr)` block logs `orderData?.name`, but `orderData` was declared with `const` *inside* the sibling `try` block — out of scope in `catch` in JS. Whenever anything in that `try` threw, referencing `orderData` in the `catch` threw a **new** `ReferenceError`, which propagated to the outer handler and was logged as `Admin API exception: orderData is not defined` — completely masking whatever the real Xero error actually was. This is why every order from #1058 (2026-07-14, the first order placed after the `9e29e65` commit) onward showed only this cryptic message in logs and 0 rows in `XeroInvoice`, while the live Xero org still had 24 real invoices through order #1057 (2026-07-01/03, before the commit).

**Fix:**
- `xero.server.js` — removed `cOGSAccountCode: COGS_ACCOUNT_CODE` from all 3 call sites and the now-unused `COGS_ACCOUNT_CODE` constant.
- `api.payment.notify.jsx` — hoisted `let orderData;` above the inner `try` block so the existing error-logging `catch` can actually reference it (and so any *future* real Xero error is visible in logs instead of masked).

**Live verification (same test order #1062, Playwright + container logs + DB):** after rebuilding/recreating the `hw-storefront-ui-node-docker` container, the ITN log for the new order showed a clean sequence with no masked exception:
