## -- Resolution (VERSION bumped 2026.07.1 → 2026.07.2)

### Collection page bug (app/errors/260708064442-errors.png open vs 260708064458-errors.png closed)
Root cause: `.collection-layout` unconditionally declared `grid-template-columns: 220px 1fr`
at >=45em. `<aside className="collection-filters">` is only rendered when `filtersOpen` is
true. With the filter closed, `.collection-results` became the grid's only child, and CSS
grid auto-placement filled the first (220px) track instead of the wide one — shrinking every
product card to sidebar width.

Fix: added a `.filters-open` modifier class toggled from React state
(`app/routes/($locale).collections.$handle.tsx`), so the 2-column template only applies when
the filter panel is present. Default (no modifier) `.collection-layout` is now
`grid-template-columns: 1fr` unconditionally (`app/styles/app.css`).

### Filter default state + floating toggle
- `filtersOpen` default flipped `useState(true)` → `useState(false)` (closed on load).
- Toggle button made "floating" via `position: sticky`. Discovered mid-verification (Playwright
  `browser_evaluate`, not just screenshots) that sticky only sticks within its *containing
  block* — the button's original parent, `.collection-header`, is a short flex row (~40-90px),
  so it stopped sticking after ~900px of scroll. Fixed by making `.collection` itself
  `display: grid` and moving the button to be a direct child with `grid-row: 1 / -1;
  justify-self: end; align-self: start` — its containing block is now the full height of
  `.collection`, so it stays pinned through the entire product grid. Verified pinned at
  `top: 16px` even at `scrollY: 10000`.

### Return-to-top button
New `app/components/ScrollToTopButton.tsx`, reusing `StickyScrollMenu.tsx`'s
`IntersectionObserver`-on-`header.site-header` reveal pattern (hidden until the site header
scrolls out of view) and matching its Tailwind FAB visual style (40×40px, `bg-black`,
`rounded-lg`, `shadow-lg`), fixed bottom-left (opposite corner from `StickyScrollMenu`'s
top-left button) so the two floating controls read as one consistent system with no overlap.
Rendered once in `app/routes/($locale).collections.$handle.tsx`.

### Cloudflare ingress (finishing todo-cloudflare.md)
Per user's explicit choice, routed the **zone apex `hoseworld.store`** (not a subdomain) to the
hydrogen storefront (port 5130) via the existing `shopify-dev` tunnel.
- `hoseworld.store` already had an A record (confirmed via `curl -I`/`dig`: registrar parking
  page, `server: cloudflare`, `x-server-powered-by: AfrRouter`, `last-modified` 2020) —
  `cloudflared tunnel route dns` refused to overwrite it by default. Re-ran with
  `--overwrite-dns` after confirming with the user what currently lived there.
- Added `hostname: hoseworld.store -> http://localhost:5130` to `~/.cloudflared/config.yml`
  above the catch-all.
- Restarted the running `cloudflared tunnel run shopify-dev` process (killed + relaunched in
  background) to pick up the config change.
- Verified: `curl -I https://hoseworld.store` → 200, and a byte-for-byte diff against
  `http://localhost:5130` is identical except the per-request CSP nonce.
- Checked `.env` for the two hostname references the prior session's `todo-cloudflare.md`
  flagged for update — both were false positives on inspection: `STOREFRONT_UI_API_URL`'s
  comment is about the storefront-ui backend URL for PayFast, correctly `app.hoseworld.store`;
  `DEV_TUNNEL_DOMAIN` is the ngrok domain registered in Shopify's Headless channel for Customer
  Account API OAuth, unrelated to the hydrogen storefront's own public URL. No `.env` changes
  made.
