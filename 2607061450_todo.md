   parity mattered).
2. **`app/lib/context.ts` did need changes**, contradicting todo_docker.md's explicit

   `workerd`-only) and took a Workers-shaped `executionContext.waitUntil`. Fixed with
   `@shopify/hydrogen`'s own `InMemoryCache` (confirmed exported at the installed version)

   pattern (`PassThrough` + `@react-router/node`'s `createReadableStreamFromReadable`).

### New errors found during verification (root cause + resolution)
All caught during actual `docker compose up` + Playwright/curl testing, not by inspection ŌĆö
this is exactly why todo_docker.md's "test implementation fully" step mattered.

1. **`SyntaxError: react-dom/server does not provide an export named 'renderToReadableStream'`**
   ŌĆö see correction #3 above. Fix: rewrite `entry.server.tsx` per Node recipe pattern.

2. **Hydrogen's own `/api/*/graphql.json` proxy route 500'd**: `TypeError: RequestInit:
   duplex option is required when sending a body.` Root cause: Node's `undici` requires an
   explicit `duplex: 'half'` on any `fetch()` call whose body is a `ReadableStream`; Workers'
   fetch has no such requirement, so Hydrogen's bundled proxy code (which forwards the
   incoming request body via `fetch()` without setting `duplex`) never hit this under Oxygen.
   Fix: a small global `fetch` shim in `server.ts` that transparently adds `duplex: 'half'`
   whenever a stream body is present and the option is missing ŌĆö patches the symptom without
   forking Hydrogen's dist bundle.

3. **Same proxy route then failed client-side with `net::ERR_CONTENT_DECODING_FAILED`**.
   Root cause: `undici`'s `fetch()` transparently decompresses gzip/br response bodies but
   leaves the original `Content-Encoding`/`Content-Length` headers on the `Response` object ŌĆö
   my `sendFetchResponse()` forwarded them verbatim, so the browser tried to gunzip
   already-plain bytes. Fix: strip `content-encoding`/`content-length` when relaying
   Hydrogen-fetched responses back through Express.

4. **The real bug, hardest to find: every cart mutation (`POST /cart`, `/cart.data`) returned
   an empty-bodied `500` with zero server-side logging** ŌĆö no exception, no `handleError`
   output, nothing, even after adding an `entry.server.tsx` `handleError` export and process-
   level `uncaughtException`/`unhandledRejection` listeners to rule out something being
   swallowed elsewhere.
   - Root cause (confirmed by reading react-router 7's own `lib/server-runtime/server.ts`
     source directly, since the failure was otherwise silent): my `createFetchRequest()`
     wired the constructed `Request`'s `AbortSignal` to `req.on('close', ...)`. Node's
     `IncomingMessage` `'close'` event fires once the request body has been **fully read**,
     not just on a genuine client disconnect ŌĆö for a small POST body that gets consumed
     almost immediately, this fired the abort signal *while the action was still running*.
     React Router's `v8_middleware` request path (which `hydrogenPreset()` enables and is
     "required for Hydrogen context") treats an aborted signal as the client having left and
     silently returns `new Response(null, {status: 500})` ŌĆö exactly the empty response
     observed, and exactly why nothing logged (react-router intentionally doesn't log
     aborted-request "errors").
   - Fix: listen on `res.on('close')` instead, and only abort if `!res.writableEnded` ŌĆö the
     correct way to detect an actual client disconnect vs. "the request body finished
     uploading."
   - This one cost the most time this session and is worth remembering: **an empty 500 with
     no server-side error output, immediately after switching a custom Node/Express adapter's
     request-body handling, is a strong signal to check the entry's `AbortSignal` wiring
     before anything else** ŌĆö it will not show up in `handleError`, `uncaughtException`, or
     `unhandledRejection`, because react-router treats it as an intentional abort, not a bug.

  adapter** ŌĆö see bug #4 above. This is the single most important thing to remember from this
