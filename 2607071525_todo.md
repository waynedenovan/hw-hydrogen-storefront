## Resolution (2026-07-07)

**Scope note:** todo.md's `INCLUDE ONLY: /hw-storefront-ui and ../hw-hydrogen-storefront`
directive conflicts with reality for this task — the reported error screenshot lives
under `hw-hydrogen-storefront-node-docker/app/errors/`, the running site in the
screenshot is served by the `hw-hydrogen-storefront-node-docker` docker container
(ngrok tunnel `alfredia-prebudgetary-nonperceptibly.ngrok-free.dev`), and today's
earlier fix (brand/sub-collection/sub-cat 3-level grouping, custom-namespace
metafields, dark-card wrapper) already lives only in that repo, not in the
non-docker `hw-hydrogen-storefront`. Treated that INCLUDE ONLY line as stale
boilerplate and fixed the bug in `hw-hydrogen-storefront-node-docker` instead —
flagging here rather than silently deviating.

**Root cause:** `app/styles/app.css` had
`.collection-section .collection-section { margin-left: 1rem; }`, applied to the
whole nested `CollapsibleSection` container div (title *and* content), not just the
title. With the page's 3-level grouping (Brand → Sub Collection → Sub-Cat
Collection), this compounded through nesting — each level's `margin-left` was
relative to its already-shifted parent, so by the 3rd level the box holding
`.products-grid` had lost ~2rem of width and was shifted right by the same amount.
Combined with `.products-grid`'s fixed `repeat(3, 1fr)` track sizing (no
`minmax(0, 1fr)`), this pushed the 3rd product column off the visible page —
matching `app/errors/260707151009-erros.png`.

**Fix:** Moved the indentation off the container div and onto
`.collection-section-toggle` (the title button) via `padding-left`, with an
explicit rule for the 2nd nesting level (`padding-left: 2rem`). The section/
sub-section title "staircase" look is unchanged; no element containing
`.products-grid` carries any horizontal offset anymore, so product cards start
flush left and use the full width of `.collection-results` regardless of nesting
depth. No JSX changes needed.

**Wrapper styling clarified with user:** todo.md's remembered values (maxWidth 80%,
rgba(80,80,80,0.65), 20px margins) conflicted with the tokens already shipped on
this page earlier today (maxWidth 90%, rgba(50,50,50,0.65), 2rem padding — per
`.claude/skills/collection-page-style/SKILL.md`). User confirmed: keep the existing
shipped tokens; only fix the width bug. No wrapper-style changes made.

**Verification:** Rebuilt the docker image (`docker compose build
hydrogen-storefront`) and recreated the container (`docker compose up -d
hydrogen-storefront`) — the container runs a pre-built production bundle with no
source bind-mount, so the CSS edit required a rebuild to take effect. Verified live
via Playwright MCP against
`https://alfredia-prebudgetary-nonperceptibly.ngrok-free.dev/collections/irrigation-pumps`:
all 3 product-grid columns now render fully within the card wrapper at desktop
width (1600px) with no cutoff, and the PASCALI ▸ JA PUMPS ▸ JAAM SUBMERSIBLE PUMPS
title staircase indent is visually unchanged from before the fix.

**File changed:** `app/styles/app.css` (`.collection-section`/`.collection-section-toggle` rules).

## Follow-up (2026-07-07, same session): two more bugs found after the section-margin fix

User re-tested at a wide viewport after the fix above and reported the grid was
*still* overlapping/exceeding the centre card — a different, unrelated cause each
time. All three fixes below required a `docker compose build hydrogen-storefront &&
docker compose up -d hydrogen-storefront` to take effect (confirmed no source
bind-mount exists for this container; editing `app/styles/app.css` alone does
nothing to the running site until rebuilt).

**Bug 2 — grid overflowed past its container even with the margin fix in place.**
`.products-grid` used `grid-template-columns: repeat(3, 1fr)` (no `minmax(0, 1fr)`).
`ProductCard`'s title uses Tailwind `truncate` (`white-space: nowrap`), whose
min-content width is the *full* untruncated title text. Without an explicit `0`
minimum, a `1fr` track can't shrink below that min-content, so the grid was forced
wider than its container instead of letting the title actually truncate. Fix:
changed both the 2-column and 3-column `.products-grid` declarations to
`repeat(N, minmax(0, 1fr))` (matching the pattern the mobile media query already
used).

**Bug 3 — this fix then revealed the results column was only 220px wide, not the
~996px it should be.** Root cause: `.collection-filters` is a semantic `<aside>`
element, which collides with a sitewide bare-tag rule (`aside { position: fixed;
right: calc(-1 * var(--aside-width)); width: min(var(--aside-width), 100vw); height:
100vh; ... }`) written for the cart/menu drawer `Aside` component.
`.collection-filters` never overrode `position`/`width`/`height`/etc., so the filter
sidebar was *always* rendered as a fixed, off-screen drawer (`left: 1920px` at a
1920px-wide viewport) — this was true before today's changes too, just invisible
because filters happened to look "closed". Losing the sidebar from grid flow left
`.collection-results` as the sole item in `.collection-layout`'s `220px 996px`
grid, and it auto-placed into the first (220px) track instead of the second. Fix:
`.collection-filters` now explicitly resets `position: static; right: auto; top:
auto; width: auto; height: auto; box-shadow: none; background: none; transition:
none;` so it stays a normal in-flow grid item. This also made the filters panel
visible/usable for the first time.

**Bug 4 (user-requested, not a defect) — very small phones showed 2 cramped
101px-wide columns.** `.products-grid` was fixed at 2 columns for the entire
`max-width: 45em` (720px) range. Added a narrower `@media (max-width: 30em)` rule
(after the existing one, so it wins by source order) dropping `.products-grid`
specifically to 1 column below 480px, while leaving the 2-column behavior in place
above that (user confirmed a ~557px-wide 2-column view already looked fine).

**Verification:** All three re-verified live via Playwright MCP against
`https://alfredia-prebudgetary-nonperceptibly.ngrok-free.dev/collections/irrigation-pumps`
after each rebuild, using both visual screenshots and
`getComputedStyle`/`getBoundingClientRect` checks on `.collection-layout`,
`.collection-filters`, `.collection-results`, `.products-grid` and `.product-card`
to confirm actual computed widths/positions, not just visual appearance. Final
state at 1920px: filters column 220px (in-flow, visible), results column 996px,
3 product columns of 316px each, no overflow. At 390px: filters stack above
results (existing `45em` breakpoint), grid is 1 column. At 557px: grid is 2
columns.

**Files changed (cumulative, this session):** `app/styles/app.css` only —
`.collection-section`/`.collection-section-toggle`, `.products-grid` (both the
`64em` breakpoint and a new `30em` breakpoint), `.collection-filters`.
