### Bug 3 — HOUSEHOLD collection (666 products) stuck "Available"/unchecked (app/errors/260714103650-errors.png)
**Real bug, root-caused and fixed — two independent, compounding causes:**

1. **Whitespace/trim mismatch (the actual "why does it look stuck" cause).** `getExistingSupplierSelections()` in `app/services/suppliers.server.js` (hw-storefront-ui-node-docker) read `userDetail.assignedFirstMajorDescription` etc. raw, without trimming — but `buildHierarchy()`/`passesSelection()` both `.trim()` the same field. The live data had 675 products stored with the literal value `"HOUSEHOLD "` (trailing space, confirmed via `hex()` on the sqlite column). The Import page's checkbox pre-check state is built directly from `getExistingSupplierSelections()`'s untrimmed output, so it compared `"HOUSEHOLD "` against the hierarchy's trimmed node name `"HOUSEHOLD"` and never matched — the checkbox rendered permanently unchecked/"Available" even though the import had already mostly succeeded (675 real products already in Shopify's `household` collection). This is why the user's repeated reselect-and-reimport looked like it did nothing: it was actually working, just never reflecting as "done" in the UI.
   - **Fix:** `getExistingSupplierSelections()` now trims all 4 dimension fields before adding to the returned sets, matching `buildHierarchy`/`passesSelection`. Regression test added (`app/services/suppliers.server.test.js`).
2. **Duplicate-collection creation (a second, independently confirmed bug, found while investigating #1).** Live Shopify Admin query found **8 duplicate "HOUSEHOLD" collections** (handles `household`, `household-1`..`household-7`). Root cause: `ensureCollectionsExist()` degrades to an *empty* title→id map whenever its `listCollections()` call hits a transient network blip (a frequent, previously-documented failure mode in this codebase) — but Shopify does not dedupe collection titles, so every run that hit that blip spawned a fresh duplicate instead of finding the existing one.
   - **Fix:** `ensureCollectionsExist()` now tracks `listCollectionsFailed` and, when set, **skips** creating any not-yet-seen title that run (logged as a `create-collection-skipped` error) instead of blindly calling `createCollection()`. Products still import normally; only the collection-create step is deferred to the next successful run. Regression test added.
   - **Live cleanup performed (Wayne approved "just delete the duplicates, I'll run the import myself"):** merged the 15 stray real products scattered across `household-2` (13) and `household-5` (2) into the canonical `household` collection (now ~690 products), then deleted the other 7 empty/near-empty duplicates via the Admin API. The actual ~666-product HOUSEHOLD import was **not** run by this session — Wayne will trigger it himself via the Import page now that the checkbox correctly reflects state and won't spawn further duplicates.
