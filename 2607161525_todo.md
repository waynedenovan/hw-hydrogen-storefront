### 2. Detailed Product page — the REAL issue finally identified: it was never fonts, it's broken gallery tiles

Your two screenshots (260716150022 / 260716150140) made the actual issue visible for the first time: a grid of broken-image tiles ("CEPEX LID" alt text) at the top of the page — **6 tiles in one screenshot, 10 in the other, a minute apart**. That variability was the diagnostic key.

Root cause (why it survived weeks of fixes on both Sonnet-5 and Fable-5): the gallery renders up to 11 *candidate* image URLs (`{id}.jpg`, `{id}_0.jpg`…`{id}_9.jpg`) without knowing which files exist, relying on each `<img onError>` hiding itself when its URL 404s. But these `<img>` tags are **server-rendered and start loading before React hydrates** — whenever a 404 lands *before* hydration finishes, the error event has already fired by the time React attaches the onError handler, so the handler never runs and the dead tile stays visible. It's a race: worst on phones (slow hydration, fast small 404 responses), which is why your phone always showed it, the tile count varied per load, and fast desktop Playwright checks kept "passing". Every prior session fixed fonts because fonts were what the tasks described — no session ever had a screenshot showing the tile grid itself until now.

The same race affected Product Cards on collection pages: a card whose local fallback image 404'd pre-hydration showed a broken image instead of its "No image" placeholder (visible in your earlier screenshot 260716141332).

Fix, two layers:
1. **Server-side (the real fix, `($locale).products.$handle.tsx` loader):** the loader now checks each gallery candidate against the actual files on disk (`fs.access` against `media/suppliers/`, same directory the `/media/suppliers/*` route serves; dynamic `node:fs` import per project pattern ERR-IMPORT-001) and only sends existing files to the client. No dead candidates are ever rendered, so there is nothing left to race — and ~10 pointless 404 requests per product page are gone too. Products with no Shopify image and no local files now show a proper "No image" placeholder instead of nothing/broken tiles.
2. **Client-side guard (`GalleryThumbnail` + `ProductCard`):** on mount, if an img is `complete` with `naturalWidth === 0` (i.e. it failed before hydration), it is treated as failed — covering the pre-hydration race for the collection-card fallback images and any file that disappears between loader and fetch.
